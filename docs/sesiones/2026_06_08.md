# Sesion Codex 2026-06-08

## Resumen

Se analizo el caso donde cada junta puede necesitar diferentes pestanas o
modulos. Se decidio implementar la configuracion a nivel de `master`, porque
es una caracteristica global de plataforma y no un dato operativo de una junta.

## Cambios realizados

- Se agrego `tenants.modules` con la lista oficial de modulos:
  - `panel`
  - `abonados`
  - `medidores`
  - `lecturas`
  - `facturacion`
  - `pagos`
  - `reportes`
  - `multas`
  - `servicios`
  - `auditoria`
  - `admin`
- Se agrego `Tenant.modulos_habilitados` como `JSONField` en la base `master`.
- Los tenants nuevos quedan con todos los modulos habilitados por defecto.
- El admin global de tenants muestra checkboxes para activar/desactivar modulos.
- `TenantPathMiddleware` bloquea con HTTP 403 el acceso directo a modulos
  deshabilitados para una junta.
- `usuarios.context_processors.roles_usuario` ahora cruza permisos por rol con
  modulos habilitados.
- El menu base y el panel principal ocultan accesos o bloques de modulos
  deshabilitados.
- Los comandos `crear_tenant` y `provisionar_tenant` aceptan `--modules` para
  crear juntas con un subconjunto de modulos desde el inicio.
- Se agrego una pantalla en el admin global de tenants para resetear la clave
  de un usuario administrador dentro de la base tenant. La accion aparece en la
  ficha del tenant como "Resetear clave admin tenant".

Ejemplo:

```bash
python manage.py provisionar_tenant rumipamba "Junta de Agua Rumipamba" \
  --admin-user admin_rumipamba \
  --admin-password "ClaveSegura123" \
  --modules panel,abonados,medidores,lecturas,facturacion,pagos,reportes,admin
```

Se ajusto la pantalla de login para mostrar la identidad de la junta:

- El lado izquierdo muestra `Sistema de Facturacion`, debajo el nombre completo
  visible construido con `nombre` + `nombre_corto`, y luego la descripcion del

  - `nombre`: `Junta Administradora de Agua Potable`

  facturas, comprobantes y reportes.
- En rutas sin tenant se usa `ConfiguracionInstitucional.nombre` si existe.

## Estado resultante

Carabuela, Rumipamba y cualquier junta futura pueden tener todos los modulos o
solo algunos, sin duplicar codigo ni compartir datos. El control queda en dos
niveles:

- El tenant define que modulos existen para esa junta.
- El rol define que puede hacer el usuario dentro de esos modulos.

## Siguientes pasos recomendados

- Definir los modulos reales de Rumipamba desde el admin global de tenants.
- Si se olvida una clave, usar el enlace "Resetear clave admin tenant" desde la
  ficha del tenant en `/admin/`.
- Migrar `master` en los ambientes donde ya exista la base:

```bash
python manage.py migrate --database=master tenants
```

- Luego continuar con la configuracion operativa de Rumipamba y el ciclo minimo
  de abonado, medidor, lectura, factura y pago.

## Continuacion: backups multi-tenant

Se adaptaron los scripts existentes de respaldo:

- `scripts/backup_db.sh` ahora puede respaldar una base especifica:

```bash
bash scripts/backup_db.sh sistema_agua_rumipamba tenant_rumipamba
```

- Se agrego `scripts/backup_all.sh` para respaldar:
  - `master`
  - `default`
  - todos los tenants activos registrados en `master`

  restaurar sobre una base temporal:
