## 6. Decisiones tecnicas tomadas y su justificacion

- Exigir rol Administrador ademas de `is_staff` para Django Admin.
  - Justificacion: Django permite por defecto entrar a cualquier usuario staff, pero la matriz del sistema limita Administracion Django al rol Administrador.
- Mantener superusuario como excepcion.
  - Justificacion: conserva el comportamiento administrativo esperado de Django.
- Separar `puede_entrar_admin_django` de `puede_ver_auditoria`.
  - Justificacion: un Administrador puede tener derecho a auditoria aunque no tenga acceso tecnico al admin.
- Implementar crear/editar abonados antes de desactivar/reactivar.
  - Justificacion: era el hueco funcional mas visible del modulo y permitia reemplazar el boton incorrecto.
- No implementar borrado fisico de abonados.
  - Justificacion: un abonado puede tener medidores, lecturas, facturas, pagos, multas, suspensiones y auditoria. Borrar fisicamente seria riesgoso.
- Validar sector/ruta en el formulario.
  - Justificacion: el modelo permitia combinaciones inconsistentes entre sector y ruta.
- Registrar auditoria en creacion y edicion de abonados.
  - Justificacion: son cambios maestros relevantes.
- Usar el patron visual de cartera pendiente para acciones de tabla.
  - Justificacion: mejora consistencia visual y evita saltos de botones en filas con acciones condicionales.
- Mantener documentacion tecnica separada por tema.
  - Justificacion: arquitectura, seguridad, permisos, pendientes y sesiones evolucionan a ritmos distintos.

## 7. Hallazgos de seguridad encontrados

- Django Admin dependia del permiso por defecto `is_staff`.
- Un usuario no Administrador podria entrar a `/admin/` si accidentalmente se marcaba `is_staff`.
- El menu podia mostrar Administracion sin verificar el requisito tecnico `is_staff`.
- El modulo Abonados no tenia CRUD desde la interfaz, por lo que la gestion podia quedar desplazada al admin.
- El listado de abonados tenia un acceso operacional incorrecto hacia creacion de medidores.
- El formulario de medidores permitia seleccionar un abonado existente, pero no resolvia la creacion del abonado.
- La relacion sector/ruta del abonado podia quedar inconsistente si no se validaba en formularios.
- El contenedor en ejecucion no montaba el codigo local, por lo que podia haber diferencias entre repo y contenedor.

## 8. Correcciones de seguridad implementadas

- Se implemento una regla explicita para acceso al Django Admin.
- Se bloqueo `/admin/` para usuarios `is_staff` sin rol Administrador.
- Se mantuvo acceso para superusuarios.
- Se alineo el menu con la regla real del backend.
- Se agregaron pruebas para evitar regresiones de permisos.
- Se agrego validacion de sector/ruta en `AbonadoForm`.
- Se restringio crear/editar abonados a Administrador y Supervisor.
- Se agrego auditoria al crear y actualizar abonados.
- Se documento el comportamiento en matriz de permisos y seguridad.
