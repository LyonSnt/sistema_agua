# Sesion Codex 2026-06-10

## Resumen

Se continuo despues de la configuracion multi-tenant y del ciclo minimo tenant.
Queda definido que `carabuela` sera el tenant principal de prueba/validacion
mientras se estabiliza el sistema; los demas tenants se agregaran despues. La
sesion se enfoco en dos frentes:

- baja/reactivacion logica de abonados;
- centralizacion del menu lateral para no modificar `base.html` cada vez que
  se agrega una opcion.

Tambien se reviso el problema de estaticos relacionado con
`menu_acciones.js`.

## Abonados activos e inactivos

Se implemento desactivar/reactivar abonado sin borrado fisico:

- Se agrego la vista `abonados:cambiar_estado`.
- La accion usa `POST` y CSRF.
- Solo Administrador y Supervisor pueden cambiar el estado.
- Se registra auditoria con acciones:
  - `DESACTIVAR`
  - `REACTIVAR`
- El listado de abonados ahora permite filtrar:
  - activos;
  - inactivos;
  - todos.
- La ficha y edicion siguen disponibles solo para abonados activos, para evitar
  que un abonado dado de baja entre en flujos operativos normales.
- En el listado se dejaron visibles las acciones principales y se movio
  desactivar/reactivar al menu de acciones `...`.

Validacion ejecutada:

```bash

docker exec sistema_agua_web python manage.py test -v 1
```

## Menu lateral centralizado

Se sustituyo el menu escrito a mano en `base.html` por una configuracion central
en `nucleo/menu.py`.

Nuevo flujo:

- `nucleo/menu.py` define secciones e items del menu.
- Cada item usa `url_name` en vez de URL fija.
- `nucleo.context_processors.menu_sidebar` construye el menu por request.
- El builder filtra items usando los permisos ya calculados por
  `usuarios.context_processors.roles_usuario`.
- Como esos permisos ya cruzan rol + modulos habilitados del tenant, el menu
  tambien queda tenant-aware.
- Las URLs se generan con prefijo tenant cuando existe
  `request.tenant_path_prefix`.
- `base.html` solo renderiza las secciones recibidas mediante el tag
  `obtener_menu_sidebar`.

Ejemplo para agregar una nueva opcion:

```python
{
    "texto": "Nuevo reporte",
    "url_name": "reportes:nuevo_reporte",
    "permiso": "puede_ver_reportes",
    "rutas_activas": ["/reportes/nuevo-reporte/"],
}
```

## Estaticos y menu de acciones

Se reviso un error al ejecutar `collectstatic`:

```text
PermissionError: [Errno 13] Permission denied: '/app/staticfiles/js/menu_acciones.js'
```

Causa:

- `/app/staticfiles` estaba con propietario incorrecto dentro del contenedor.
- Django corre como `appuser`, pero los archivos estaban como `root`/otro
  propietario por el montaje de desarrollo o una ejecucion previa.
- Tras corregir permisos y ejecutar `collectstatic`, hizo falta reiniciar el
  servicio web para que WhiteNoise recargue la lista de archivos estaticos.

Correccion aplicada:

```bash
docker compose exec -u root web chown -R appuser:appuser /app/staticfiles
docker compose exec web python manage.py collectstatic --noinput
docker compose restart web
```

Validacion:

- `collectstatic`: 128 archivos copiados, 9 sin cambios.
- Logs posteriores:

```text
GET /static/js/menu_acciones.js HTTP/1.1" 200
```

## Notas sobre acciones en listados

`menu_acciones.js` no decide cuantas acciones se muestran. Solo cierra menus
`details.menu-acciones` cuando el usuario hace clic fuera.

La regla visual queda en las plantillas:

- dejar visibles las acciones principales;
- mover acciones secundarias o delicadas al menu `...`;
- mantener `POST` y CSRF para acciones que modifican datos.

Pendiente opcional:

- Extraer un include/componente para reutilizar el menu `...` si se repite en
  mas listados.
